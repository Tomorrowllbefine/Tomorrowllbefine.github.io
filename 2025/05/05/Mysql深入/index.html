filtered 字段：

这是因为在成本模型里，MySQL 需要把“通过索引筛出的行数”跟“实际满足所有条件的行数”分开来估算，才能更准确地预测整个连接操作的开销。具体来说：

  1. 全表扫描时，没有预先利用任何索引对数据进行缩小，所有的过滤条件（WHERE 或者 ON 上的每一个表达式）都要在一次完整的表扫描里一并应用。
    • 因此，扇出（fan-out）估算——也就是驱动表每读到一行后，会产生多少条“有效”行去访问下一个表——必须基于“所有条件”共同作用下的总过滤率。
    • 换句话说：你先扫整张表，然后把所有条件都当作一个整体去过滤，留下多少行，就用多少行去驱动下一个表。
  2. 索引扫描时，查询优化器先挑选出“用到那个索引”的那部分过滤条件（比如 col=常量、col BETWEEN ... 等），在索引层面就把大部分不相关行给踢掉了。
    • 那么扇出的第一步，就是“每个索引命中”会对应一次对主表行或二级索引行的访问。这个索引命中数由索引自身的基数（cardinality）和索引条件的选择性（selectivity）来估算。
    • 剩下的条件（那些没有用到这个索引的过滤表达式，比如其他列上的 b>5 AND c<10）只能在取回行之后再做过滤。所以，扇出估算要分两步：
      1. 索引第一步筛：估算出“有多少行会走到这里”（索引返回的行数）。
      2. 残余条件再筛：在这部分行上再应用“其他条件”，估算最后剩下多少行。

如果把这两部分混在一起估算，优化器就不知道究竟是哪个条件在“扮演”先过滤、哪个在“后过滤”，从而很难精确计算“到底要访问下一个表多少次”。把它们拆开，就是所谓的 condition filtering：

  • 索引条件：决定了“多少次磁盘/内存查索引”；
  • 残余条件：决定了“在这些索引命中基础上，还能再砍掉多少行”。

这样一来，无论是全表扫描（一次性全部过滤）还是索引扫描（先索引、后过滤），优化器都能用最贴近真实的过滤率去估算连接驱动表的扇出成本，从而选出真正代价最低的执行计划。

【调节磁盘与CPU速度差异】InnoDB 的 Buffer Pool

Topic 1：缓存的必要性

对于使用InnoDB作为存储引擎而言，不管是存储用户数据的索引，还是各种系统数据，都是以“页”的形式存储在“表空间”中的。而所谓的“表空间”实则是InnoDB引擎对文件系统上的若干实际文件的抽象，即数据本质上还是存储在了磁盘上。

此外，InnoDB引擎在处理客户端的请求时，就会把完整的“页”的数据全部加载到内存中，这即是说只要访问了某一页中的一条记录，那么该页的所有数据都需要加载到内存中。

那么，速度慢了CPU很多个数量级的磁盘与前者之间如何进行匹配呢，这就需要其间缓存的相关支撑。将当前待查询记录的页加载到内存中后就可以对其进行读写操作，在进行完读写访问后并不急着将其释放，而是缓存起来，那么将来有请求再次访问时，就可以省去磁盘的IO开销。

Topic 2：InnoDB 的 Buffer Pool

01：Buffer Pool

在InnoDB存储引擎中，上述缓存技术的实现即是 “Buffer Pool”，设计者为了缓存磁盘中的页，在Mysql服务器启动时就向操作系统申请了一片连续的内存，命名为Buffer Pool。默认大小为128M，可以在启动配置文件中配置参数 innodb_buffer_pool_size 的值，如下：

具体大小取决于机器配置以及实际需要，若有512G内存，分配个几百G作为Buffer Pool也可以；

02：Buffer Pool 内部组成

在Buffer Pool中缓存的缓存页大小与磁盘上的一致，都是16KB。为了更好的管理这些在Buffer Pool中的缓存页，设计者为每一个缓存页都创建了一些所谓的控制信息，这些控制信息包括该页所属的表空间编号、页号、缓存页在Buffer Pool中的地址、链表节点信息、一些锁信息以及LSN信息。

每个缓存页对应的控制信息占用的内存大小是相同的，每个页对应的控制信息占用的一块内存称为一个控制块，控制块和缓存页是一一对应的，它们都被存放到 Buffer Pool 中。其中控制块被存放到 Buffer Pool 的前边，缓存页被存放到 Buffer Pool 后边，分配控制块及缓存页后不足以组成新的块的内存空间就是碎片，整体如下图：

每个控制块大约占用缓存页大小的5%，在MySQL5.7.21这个版本中，每个控制块占用的大小是808字节。而我们设置的innodb_buffer_pool_size并不包含这部分控制块占用的内存空间大小，也就是说InnoDB在为Buffer Pool向操作系统申请连续的内存空间时，这片连续的内存空间一般会比innodb_buffer_pool_size的值大5%左右。

03：空闲页 - free 链表管理

在MySQL服务器启动后，会根据申请到的连续空间进行划分，一部分属于控制块区，一部分属于缓存页区。当请求查询某一条记录而将磁盘中对应页的数据装载入缓存时，如何确定要将这个页存储在缓存页的哪个位置呢？或者换个角度理解，如何区分Buffer Pool中哪些缓存页是空闲、哪些是已使用呢？

此时，引入一个free 链表，所有空闲的缓存页对应的控制块都作为该链表的一个节点。那么在刚刚完成初始化的Buffer Pool中所有的缓存页都是空闲的，所以每一个缓存页对应的控制块都会被加入到free链表中，假设该Buffer Pool中可容纳的缓存页数量为n，那增加了free链表的效果图就是这样的：

为了管理好这个free链表，特意为这个链表定义了一个基节点，里边儿包含着链表的头节点地址，尾节点地址，以及当前链表中节点的数量等信息。

这里需要注意的是，链表的基节点占用的内存空间并不包含在为Buffer Pool申请的一大片连续内存空间之内，而是单独申请的一块内存空间。

链表基节点占用的内存空间并不大，在MySQL5.7.21这个版本里，每个基节点只占用40字节大小。后边我们即将介绍许多不同的链表，它们的基节点和free链表的基节点的内存分配方式是一样一样的，都是单独申请的一块40字节大小的内存空间，并不包含在为Buffer Pool申请的一大片连续内存空间之内。

在引入了free链表后，每当需要从磁盘中加载一个页到Buffer Pool中时，就从free链表中取一个空闲的缓存页，并且把该缓存页对应的控制块信息（该页所在的表空间、页号之类的信息），然后把该缓存页对应的free链表节点从链表删除，即表示该页已经被使用。

04：缓存命中 - 缓存页的哈希处理

当我们需要访问某个页中的数据时，就会把该页从磁盘加载到Buffer Pool中，如果该页已经在Buffer Pool中的话直接使用就可以了。那么问题也就来了，我们怎么知道该页在不在Buffer Pool中呢？难不成需要依次遍历Buffer Pool中各个缓存页么？

回顾一次 SELECT 查询的过程，在找到对应记录所在的聚簇索引B+树的叶子结点页号，比如确定页号为 page_no = 12345，所在表空间为 space_id = 67。也就是说，我们其实是根据 表空间号 + 页号 来定位一个页的，也就相当于表空间号 + 页号是一个key，缓存页就是对应的value。

所以我们可以用**表空间号 + 页号作为key，缓存页(对应控制块的指针)**作为value创建一个哈希表，在需要访问某个页的数据时，先从哈希表中根据表空间号 + 页号看看有没有对应的缓存页，如果有，直接使用该缓存页就好。如果没有，那就从free链表中选一个空闲的缓存页，然后把磁盘中对应的页加载到该缓存页的位置。

Q：为什么不是直接映射到内存页地址？
  • 需要维护大量元数据（LRU、Free、Flush、状态标记等），这些都存储在控制块中；
  • InnoDB 的许多操作（淘汰页、刷盘、链表维护）都围绕控制块进行；
  • 内存页本身是比较「哑的」，控制逻辑都在控制块里；
Q：怎么理解三种链表的底层实现？

这些链表（如 LRU、Free、Flush）中的「节点」是抽象出来的，而节点之间的联系就是靠控制块（Control Block）中的前驱、后继指针字段来维护的。

  • 它不是一个专门的数据结构对象（如 List<Page>），而是由 多个控制块通过指针互相链接形成的双向链表。

  • 每个控制块中都维护两个字段：

    • prev（前驱控制块指针）；
    • next（后继控制块指针）；

05：脏页 - flush 链表管理

若用户对Buffer Pool中的某个页中的记录进行修改，此时对于该页而言，存在缓冲池版本与磁盘中的版本的数据不一致情况，该页即为“脏页”。

以用户修改为最新数据，那么必须要将更新的内容同步回磁盘（刷新），同步的策略目前可想到的如下：

  • 修改缓冲池中的一个页后，立即将其同步刷新到磁盘；

频繁的往磁盘中写数据会严重的影响程序的性能（毕竟磁盘慢的像乌龟一样）；

  • 修改缓冲池中的一个页后，立即将缓冲池中所有页同步刷新到磁盘；

假如Buffer Pool被设置的很大，比方说300G，那一次性同步这么多数据势必会耗费大量的时间，在请求量高时对性能有很大的影响；

对策：创建一个存储脏页的链表，凡是修改过的缓存页对应的控制块都会作为一个节点加入到一个链表中，因为这个链表节点对应的缓存页都是需要被刷新到磁盘上的，所以也叫flush链表。链表的构造和free链表差不多，假设某个时间点Buffer Pool中的脏页数量为n，那么对应的flush链表如下：

06：缓存空间维护 - LRU链表的管理

缓存不够问题：

Buffer Pool对应的内存大小毕竟是有限的，如果需要缓存的页占用的内存大小超过了Buffer Pool大小，后续查询的页则无法进行缓存（基于局部性原理而言，缓存命中率在缓冲池满后会急剧下降）。因此需要把某些旧的缓存页从Buffer Pool中移除，然后再把新的页放进来，维护缓存空间。

不难理解，在缓冲池将要满或者满的情况下，要保留的是最近频繁使用的缓存页，剔除最近很少使用的缓存页。

版本一：简单的LRU链表：

管理Buffer Pool的缓存页其实也是这个道理，当Buffer Pool中不再有空闲的缓存页时，就需要淘汰掉部分最近很少使用的缓存页。

因此，设计者提出构建**LRU链表**，通过如下的规则来处理节点以实现上述的解决方案：

  • 首先，每个链表节点对应一个缓存页的控制块；
  • 当对应的缓存页被访问，如果该页不在Buffer Pool中，在把该页从磁盘加载到Buffer Pool中的缓存页时，就把该缓存页对应的控制块作为节点塞到链表的头部；
  • 如果该页已经缓存在Buffer Pool中，则直接把该页对应的控制块移动到LRU链表的头部；

也即是说：只要我们使用到某个缓存页，就把该缓存页调整到LRU链表的头部，这样LRU链表尾部就是最近最少使用的缓存页；当Buffer Pool中的空闲缓存页使用完时，到LRU链表的尾部找缓存页淘汰；

LRU(Least Recently Used，最少最近使用)原则，筛选出最近一段时间内，使用/访问频率最少的item；

版本二：划分区域的LRU链表：

简单LRU链表使用过程中会面临两方面的问题：

  1. InnoDB提供了一个看起来比较贴心的服务——预读（英文名：read ahead）。所谓预读，就是InnoDB认为执行当前的请求可能之后会读取某些页面，就预先把它们加载到Buffer Pool中。根据触发的方式，可细分为：

    • 线性预读：依据系统变量innodb_read_ahead_threshold，如果顺序访问了某个区（extent）的页面超过这个系统变量的值，就会触发一次异步读取下一个区中全部的页面到Buffer Pool的请求；

      这个innodb_read_ahead_threshold系统变量的值默认是56，我们可以在服务器启动时通过启动参数或者服务器运行过程中直接调整该系统变量的值，不过它是一个全局变量，注意使用SET GLOBAL命令来修改；

    • 随机预读：如果Buffer Pool中已经缓存了某个区的13个连续的页面，不论这些页面是不是顺序读取的，都会触发一次异步读取本区中所有其的页面到Buffer Pool的请求；

      依据innodb_random_read_ahead系统变量，它的默认值为OFF，也就意味着InnoDB并不会默认开启随机预读的功能，若开启可以通过修改启动参数或者直接使用SET GLOBAL命令把该变量的值设置为ON；

    问题：预读到Buffer Pool中的页成功的被使用到，那就可以极大的提高语句执行的效率。但反之，如果大量的将来不会被访问的页都插入LRU的头部，同时若缓冲池的大小较小时，很可能导致原本处在LRU头部（访问频率高很需要缓存）的缓存页被挤兑至链表末尾并很快被淘汰，这就是劣币驱逐良币，会大大降低缓存命中率。

  2. 对一些需要执行扫描全表的查询语句（比如没有建立合适的索引或者压根儿没有WHERE子句的查询），会导致Buffer Pool中的所有页都被换了一次血，使得其它查询语句在执行时又得执行一次从磁盘加载到Buffer Pool的操作，严重地影响到其他查询对 Buffer Pool的使用，从而大大降低了缓存命中率。

总结上述两种情况：

  • 加载到Buffer Pool中的页不一定会被用到；
  • 如果非常多的低频使用率的页被同时加载到Buffer Pool时，可能会把那些使用频率非常高的页从Buffer Pool中淘汰掉；

因此，设计者提出一种方案，将LRU链表按照一定的比例分成两个区域（ young 区和 old 区）：

  • young区（热数据）：存储高频使用率的缓存页；
  • old区（冷数据）：存储低频使用率的缓存页；

随着程序的运行，某个节点所属的区域也可能发生变化，默认情况下，old区域在LRU链表中所占的比例是37%，也就是说old区域大约占LRU链表的3/8。
